Architecture · Level 1

How ANet thinks.

A friendly walkthrough of the core engine — how it takes one message you type and turns it into a coordinated plan, executed by many specialist agents working together. No jargon, just diagrams and a real example at the end you can follow line-by-line.

Scroll
The big picture

One request flows through four stages.

Every message you type travels the same four-stage pipeline. This is the single most important thing to learn — once you see it, the rest of ANet is just "who plays which role here." Think of it as an assembly line: each stage has one job, hands off cleanly to the next, and the line is short enough to picture end to end.

The pipeline lives in anet/core/engine.py. It is plain Python — no LangChain, no graph framework — about 700 lines, four stages.

What each stage does

The four roles, one at a time.

1

Planner

The Manager LLM

Reads your request and either replies directly (greeting, trivial question) or emits a DAG — a list of steps, each labelled with the agent that will do it and what it depends_on.

2

Executor

asyncio.gather

Walks the DAG. Every step whose dependencies are done runs in parallel. Each step boots its own agent, with its own model and its own tools.

3

Checker

A second LLM judges

Labels each result success, partial, or failure. On partial it can write an adjustment — a tweak to the plan — and the executor retries that branch.

4

Synthesizer

Streams to your terminal

Combines every step's result into one coherent reply, streamed token-by-token. This is the only stage you actually see typing back at you.

The mental shortcut: Planner says what, Executor says do it now, Checker says was that good enough?, Synthesizer says here's the answer.
Zoom in

Inside one step: the agentic loop.

When the Executor runs a step, it hands control to the orchestrator (anet/core/orchestrator.py). The orchestrator is what actually talks to a model and tools — over and over — until the agent decides it's done or hits a safety limit.

  1. The agent's model proposes a tool call. Each agent is given only its own toolbox (research gets web tools, code gets file/shell, etc.).
  2. The safety gate inspects it. Writes (shell commands, edits, destructive file ops) pause for your y / n / a. Reads pass through.
  3. The tool runs and returns a result. Could be 50ms (a file read) or 30 seconds (a web crawl).
  4. The result is appended to the conversation and the model is called again. It may call another tool, or it may answer.
  5. The loop ends when the model stops calling tools — or when it hits the agent's max_steps cap, or when the cycle detector spots the same write repeated three times.
How the pieces wire up

The whole core, on one page.

Five modules do almost all the work. Everything else in anet/core/ is a loader (config, ExTools, MCP servers) or a store (sessions, memory, skills).

Read the diagram as: your terminal talks to the Engine (pipeline); the Engine delegates each step to the Orchestrator (the loop); the loop calls tools and MCP servers; everything is persisted in three stores at the bottom that any session can read on a future run.

Two kinds of memory

What it remembers, and when.

Memory is the only piece of state ANet carries between requests. There are two layers — both fully local — and they serve different jobs.

S

Short-term — rolling context

in conversations.db

The recent turns of the current session, token-budgeted. When the conversation overflows the budget, older turns are summarised into a single rolling summary; the most recent turns are always kept verbatim.

L

Long-term — mem0

~/.anet/memory/

Every five turns (and once on clean exit), recent conversation is handed to mem0 — which uses your LLM to extract durable facts about you and stores them in a local Chroma vector DB. Surfaced again next session.

The key insight: long-term memory isn't just a transcript dump. Mem0 extracts, de-duplicates, and classifies each fact, so the store stays small and useful instead of growing forever.
Why it doesn't run away

Safety lives in three places.

Confirmation gate

Every shell command, every edit, every destructive file op pauses for y / n / a. Reads pass through silently.

Per-agent step cap

Each agent has a max_steps. The agentic loop refuses to run beyond it — no runaway loops, no surprise bills.

Cycle detection

If the same write is attempted three times in a row, the loop stops. The model can read in a tight loop, but not write in one.

A worked example

Let's trace one real request end to end.

Enough theory. Here is a single, slightly ambitious request and exactly what ANet does with it. Read this one section and you'll see every piece from above doing its actual job.

You type into the terminal:

"Search the web for the top 5 most starred Python repositories on GitHub right now. Write a Python script that fetches their README (first 500 chars) via the GitHub API and saves a report.json with name, stars, and readme snippet for each. Write a test that validates the JSON structure and that all 5 entries exist. Run it, fix any failures, re-run until clean. Then send the final report.json contents as a formatted message to a Telegram bot."

That's one sentence to a human, but four very different jobs: research the web, write code, test and fix that code, deliver a message to a third-party service. No single agent is good at all four — and that's the whole point of ANet.

Step 1 The Planner reads it.

The Manager LLM looks at your message and recognises three distinct skill-shapes: search the web, write & test code, send a chat message. It picks the agent that owns each skill and lays them out in dependency order — code can't run until research returns the repo list, Telegram can't run until the report file exists. So the plan ends up purely sequential this time. (If you'd said "research X and research Y," the Planner would have run them side-by-side instead.)

Step 2 The DAG it builds.

Three real steps, plus the Checker sitting like a gate between B and C. The yellow self-loop on B is the agentic loop you saw earlier — the code agent iterating on its own to fix its tests. The orange arrow is the Checker's retry path, used only if B can't get itself clean in the time it has.

Step 3 The Executor runs the DAG.

A

research_agent runs research_agent

Calls web_search("most starred Python repositories GitHub"), opens the first credible result with web_fetch, and parses out the five top names and their star counts.

Returns: a short JSON list of {name, stars}. Hands control back to the Executor.

B

code_agent runs code_agent

Receives the repo list. Then loops on its own (this is the inner agentic loop):

  1. Drafts fetch_report.py with edit_tool — the safety gate pauses for y/n/a; you press y.
  2. Drafts test_report.py — gate fires; y again.
  3. Runs pytest via shell_tool — gate fires; y.
  4. One test fails — readme snippet was too long. Reads its own file with grep_tool, edits the slice to readme[:500], re-runs. All green.

Returns: "report.json written, all tests passing." Stops calling tools, so the loop ends.

⟳

checker_agent inspects step B checker_agent

Reads what B claimed and verifies it briefly. Tests passed, file exists, structure looks right → labels it success. The DAG advances to C.

If the tests were still red after a few attempts, the Checker would stamp partial and send the Executor back to B with a hint — that's the orange arrow in the diagram above. The whole loop is bounded so it can't churn forever.

C

tele_agent runs tele_agent · external

This one isn't built-in — it comes from your pack (a custom agent you or someone else wrote). Reads report.json, formats it as a Markdown message, and calls tele_tool.send_message(...) with your bot token.

Result: the message lands in your Telegram chat. Returns "sent."

Step 4 The Synthesizer wraps up.

With every step labelled success, the Synthesizer takes the combined results and composes one short reply for your terminal — confirming what ran, what got written, and that Telegram was notified. It streams in token-by-token so you see it typing live.

ANet replies in your terminal:

Done. Found the top 5 starred Python repos (public-apis/public-apis, vinta/awesome-python, …). Wrote fetch_report.py and test_report.py into your project. One test failed initially (readme snippet length); fixed the slice and re-ran — all green. report.json is saved at the project root, and the formatted summary was sent to your Telegram bot.

What you just saw: the Planner picked agents, the Executor coordinated them, each agent ran its own loop with tools, the safety gate held back every write, the Checker validated the important hand-off, and the Synthesizer spoke. That's all of Level 1 in a single request.
Level 1 recap

If you remember five things, you've got it.

  1. The pipeline has four stages: Planner → Executor → Checker → Synthesizer. They live in engine.py.
  2. The Planner emits a DAG. Independent steps run in parallel; dependent steps wait.
  3. Each step is an agent running a loop: model → tool → result → model again, capped by max_steps.
  4. Writes pause for confirmation at the safety gate; reads don't.
  5. Two memory layers — rolling short-term inside a session, fact-extracted long-term across sessions.

That's the whole core in one page.
