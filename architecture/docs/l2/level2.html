Phase 1 · Decompose

What the decomposer actually outputs.

Decompose is one model call, with one job: turn your natural-language message into a strictly-typed JSON object. Two shapes are valid — a trivial fast-path reply, or a set of annotated subtasks. Anything else is a bug. The code lives in anet/core/AdaptOrch/decomposer/. 

Decomposer output — schema// Shape A — fast-path direct reply (greeting, single fact, one-liner)
{
  "trivial": true,
  "reply":   "<assistant's answer>"
}

// Shape B — a set of subtasks → dag.build() turns these into G_T
{
  "trivial":       false,
  "request_class": "coding",   // coding | reasoning | rag | general
  "subtasks": [
    {
      "id":               "s1",
      "description":      "<what to do, self-contained>",
      "agent":            "<agent name from the list>",
      "depends_on":       [],           // e.g. ["s1","s2"] → wait for those subtasks
      "complexity":       "medium",     // low | medium | high  → cost weight w_i
      "estimated_tokens": 500,          // preferred over complexity when present
      "coupling":         "strong"      // none | weak | strong | critical → c(u,v)
    }
  ]
}

Every field is load-bearing. depends_on becomes the DAG's edges (and drives routing). complexity/estimated_tokens maps through COMPLEXITY_WEIGHT into each node's cost weight w_i (feeding depth δ). coupling maps through the COUPLING table (none=0, weak=0.3, strong=0.7, critical=1.0) into the edge weight c(u,v) that feeds coupling density γ. The decomposer never executes — it only plans.

Why JSON-only? The decomposer prompt forbids prose and says "MAXIMIZE PARALLELISM: add a depends_on entry ONLY when one subtask's output is genuinely an input to another." Everything else stays [] so it can run concurrently. That single rule is why the DAG's shape — and therefore the topology — reflects the request's real structure instead of an arbitrary chain.

Prompt anatomyWhat the decomposer sees.

The messages are assembled fresh each turn by build_prompt() from these pieces, in this order:

  1. The decomposer system prompt. "You decompose a user request into a DAG of subtasks and route each to an agent. You never execute — only plan." Plus the fast-path, parallelism, coupling, and tool-complete-routing rules.
  2. Relevant context from memory. RecMem hits (standing preferences + task-relevant facts) prepended to the user message when non-empty — this is how a preference like "prefix functions with anet_" reaches a coding subtask.
  3. The task template. "Analyze the following task and decompose it… Task: {T}".
  4. AVAILABLE AGENTS list. Each agent's name + its declared task_types — this is how TOOL-COMPLETE routing works: each subtask goes to the single agent that can finish it end-to-end with its own tools.
  5. The output spec. The exact JSON above, "Return ONLY this JSON (no prose)."

Most of the prompt is fixed text (and marked cacheable when the provider supports prompt caching), but the memory block changes every turn — which is how a standing preference actually reaches a subtask even when you don't mention it. The decomposer reasons over the rolling summary + verbatim recent turns, not the summary alone.

Phase 2 · DAG

The graph, and the four numbers it computes.

dag.build() turns the decomposer's subtasks into a formal task DAG G_T = (V, E, w, c) — pure Python, no LLM, all in O(|V|+|E|). It then computes the structural properties the router needs. These four numbers are the routing input.

MetricWhat it measuresHow
ω (omega)Parallelism width — how many subtasks can run at once.Max layer size across the topological (Kahn) layering.
δ (delta)Critical-path depth — the longest weighted dependency chain.Single DP pass over node cost weights w_i.
γ (gamma)Coupling density — how much context the edges share.Σ c(u,v) / |E| (0 when there are no edges).
layersTopological layering — used by the hybrid executor.Kahn's algorithm; also yields r = ω/|V|.
Why the weights matter: a declared dependency with no stated coupling defaults to strong (0.7) — an edge exists precisely because one subtask feeds another. High-complexity nodes weigh 9.0 vs 1.0 for low, so δ reflects real effort along the chain, not just node count.
Phase 3 · Route · Algorithm 1

How the DAG's shape picks a topology.

Routing is deterministic and O(1) given a built DAG — no LLM. It reads ω, γ, |V|, |E|, and r = ω/|V|, then falls through five rules ("Algorithm 1") to one of four canonical topologies. Code lives in anet/core/AdaptOrch/router/. 

router — Algorithm 1# thresholds default to θω=0.5, θγ=0.6, θδ=5 (overridable in orchestration:)
if   |E| == 0:                          return τ_P   # parallel: no dependencies at all
elif ω == 1:                          return τ_S   # sequential: a single chain
elif γ > θγ and |V| > θδ:            return τ_H   # hierarchical: coupled + many subtasks
elif r > θω and γ <= θγ:            return τ_P   # parallel: wide, low coupling
else:                                return τ_X   # hybrid: parallel within layers, sequential between
TopologyFires whenExecutor behaviour
τ_P parallelNo edges, or wide + low coupling (r>θω, γ≤θγ).All subtasks run at once; independent by design.
τ_S sequentialω=1 — a single dependency chain.One subtask at a time, each fed its predecessor's output.
τ_H hierarchicalγ>θγ and |V|>θδ — coupled + many subtasks.A coordinator delegates tightly-coupled work.
τ_X hybridEverything else (mixed width and coupling).Parallel within each layer, sequential between layers.
It leaves a trail. Every decision — the DAG features plus the chosen topology — is appended to orchestration/routing_log.jsonl. That is the (features → topology) dataset for future learned routing, collected from the very first decision rather than after the fact.
Phase 4 · Execute

How the chosen topology actually runs.

Each topology has its own executor in anet/core/AdaptOrch/executors/ (parallel.py, sequential.py, hierarchical.py, hybrid.py). They differ only in ordering and what context each subtask sees; every subtask ultimately runs through the same shared agent loop, orchestrator.run.

  1. Parallel (τ_P) runs every subtask concurrently — they were declared independent, so no output needs to reach another.
  2. Sequential (τ_S) runs the chain one subtask at a time, threading each direct predecessor's output into the next subtask's context.
  3. Hierarchical (τ_H) hands tightly-coupled work to a coordinating agent that delegates and integrates.
  4. Hybrid (τ_X) walks the DAG's topological layers: it runs everything in a layer in parallel, then passes the whole prior layer forward before starting the next.
What isn't here anymore: there is no execute-then-check retry loop. AdaptOrch has no success/partial/fail gate that bounces a subtask back. A subtask iterates internally (its own agent loop), and if the final outputs genuinely conflict, that is handled once in synthesis — not by re-running the whole graph.
Inside one step · the agentic loop

The Orchestrator — what one step really does.

Whatever the topology, each subtask is run by orchestrator.run(...) — unchanged from before AdaptOrch. This is a tight loop: ask the agent's model what to do, intercept any tool call, check the safety policy, run the tool, feed the result back. It sounds simple — the details are where the safety lives.

2.1The confirmation policy table.

Not every tool call asks for approval — that would make ANet unusable. The orchestrator carries a small policy in _CONFIRM_TOOLS that says, per tool, exactly which actions need a y / n / a:

ToolWhich actions need approvalWhy
shell_toolevery commandArbitrary shell — always dangerous.
code_executionevery runRuns arbitrary Python.
edit_toolevery editModifies your files.
download_fileevery downloadWrites a file to disk.
file_toolwrite_file, create_folder, delete_file, copy_file, move_file, rename_file, zip_files, unzip_fileThe read actions (list_dir, read_file) pass through.
open_applaunch_and_type, type_text, click_element, keyboard_shortcutChanges desktop state. Plain "open" is silent.
memory_toolclear, deleteNever wipe memory silently.

Read-only operations are explicitly listed (_READ_ONLY_OPS, _READ_ONLY_ACTIONS) so the cycle detector knows to exempt them. The model can list a directory in a tight loop if it wants; it just can't write in one.

2.2Provider dispatch — agent_runner.py.

Every agent declares its own model and provider. agent_runner normalises that into a unified call. Two paths:

OpenAI-compatible

openrouter · openai · google · vertex_google

Uses AsyncOpenAI with a per-provider base_url. OpenRouter routes 300+ models. Gemini's "thought signature" extras are preserved verbatim in _message_to_dict so thinking models don't reject their own looped-back messages.

Anthropic native

anthropic · vertex_anthropic

Uses anthropic.AsyncAnthropic. Tool-call shape is translated to match the OpenAI shape the orchestrator already speaks — one common return type either way.

2.3Memory injection — every subtask, every time.

Before the model call, a short "RELEVANT MEMORY" block is built from the long-term store (RecMem by default) by pulling:

  • Standing memories — always_inject categories scoped to this agent (your preferences, your identity). Surfaces even when there's no keyword overlap. Works on both RecMem and mem0.
  • Task-relevant facts — semantic search against the task, so "write a function" doesn't drag in every memory mentioning code.

Snippets are length-capped so no single mega-memory can flood a small model's context. The block is empty (and omitted entirely) when nothing is relevant — the common case with RecMem, since most turns never consolidate a durable fact at all.

Validation, without a checker stage

Where "good enough" actually gets decided.

AdaptOrch has no checker phase and no automatic success/partial/fail retry loop. Validation happens in two places instead — and neither one bounces the whole graph back.

Inside a subtask

the agent's own loop

A subtask iterates on itself — code_agent runs its tests, reads the failure, edits, re-runs, until clean or it hits max_steps. That loop is the agent's, bounded by the cycle detector. No outer stage grades it.

As a validation subtask

checker_agent in the DAG

checker_agent still exists — as an agent. When the decomposer wants a result verified, it emits a checker_agent subtask that depends_on the work. Its verdict is just another subtask output that flows forward into synthesis.

The demoted signal: the consistency score (CS) that older designs used to gate the whole merge is now just an internal signal used inside the vote and resolve operators. It no longer decides whether synthesis merges or arbitrates — synthesis dispatches by topology and request class instead (next section).
Phase 5 · Synthesize · Algorithm 2

One coherent reply, from five operators.

With every subtask done, synthesis composes the final user-facing reply. It's the only phase that writes plain prose (every other phase speaks JSON or numbers). But "compose" is only one of five ways to merge — select_operator() picks the right one from the execution topology and the request class. Code: anet/core/AdaptOrch/synthesizer/.

OperatorSituationWhat it does
composeDistinct complementary parts (backend / frontend / tests).Integrate every part into one deliverable — keep all, don't pick.
aggregateResearch / evidence (findings A, B, C).Unify into one summary; note disagreements rather than choosing.
voteRedundant attempts at the SAME answer.Majority wins (this is where CS is used as a signal).
rankN competing candidates for one goal.Rank by quality, return the single best.
resolveA genuine contradiction on the same point.Arbiter decides (prefers fresh evidence), with a bounded re-route loop.

By default, distinct-subtask topologies (ANet's normal case) pick compose, or aggregate for rag/research requests. A sequential chain resolves to sequential_last (the chain's last output already carries the earlier ones), and a single subtask to single. vote/rank/resolve are only chosen via an explicit decomposer hint or a declared-redundant decomposition — so parallel subtasks that are distinct by design never trigger a spurious arbiter.

  1. Input: the subtask outputs (with agent, task, output) plus the original user message and the chosen topology.
  2. Select: select_operator() maps topology + request class (+ any hint) to one of the operators above.
  3. Combine: a single terminal LLM call runs the operator's prompt over the numbered outputs — or, for vote/sequential_last/single, the answer is emitted directly.
  4. Stream: tokens are printed live through the token sink — no perceived "stuck" gap between the last agent finishing and the reply appearing.
  5. Persistence: the reply is stored in conversations.db under the thread, ready for next turn / --resume.
The resolve re-route: only resolve can loop. On a genuine contradiction it re-routes with an inflated coupling estimate (γ' = γ + 0.2) so the graph re-runs more tightly-coupled — but the loop is bounded. This is the one place AdaptOrch ever revisits execution, and it fires on a real conflict, not on ordinary distinct outputs.
Where state lives

One DB, one summary, one vector store.

Three things outlive a single request. Knowing where each lives makes session switching, --resume, and memory recall stop feeling like magic.

conversations.db

sessions/conversations.db

One shared SQLite file, accessed via aiosqlite. Rows are keyed by thread — a session is just a thread name. Switching /session is a string change, not a DB swap.

Rolling summary

engine_base + same DB

Each thread has a single text summary covering everything older than the recent verbatim window, maintained by BaseEngine._maintain_summary. Recomputed (one LLM call) only when the window overflows recent_tokens — typically once every several turns, not every turn.

RecMem + Chroma

~/.anet/memory/recmem/

Long-term facts, the default backend. A native 3-tier recurrence store — subconscious → episodic → semantic — over a local Chroma DB with on-device fastembed embeddings. mem0 is still selectable (memory.backend: mem0).

The recurrence ideaWhy most turns cost zero memory tokens.

Where a flat store would spend an LLM to extract facts on a schedule, RecMem buffers cheaply and only consolidates when something recurs:

  1. Subconscious. Every turn drops a raw snippet into a buffer, embedded on-device (fastembed) — no LLM. This is the cheap default path for almost every turn.
  2. Episodic. When an incoming interaction finds enough semantically-similar snippets already buffered (recurrence_threshold hits at sim_threshold cosine), the LLM is invoked to consolidate them into an event summary.
  3. Semantic. Recurring episodes distil into durable facts, de-duplicated (dedup_threshold) so near-identical facts don't pile up.
The load-bearing consequence: the expensive LLM consolidation runs only on recurrence, so a one-off turn costs zero memory tokens and the store stays small and useful instead of growing forever. Standing preferences (always_inject categories) still surface into prompts on both RecMem and mem0.
The constants that bound everything

Five numbers that hold the line.

ANet's safety isn't policy text — it's hard caps in code, low enough that a runaway loop hits one of them in seconds, not minutes.

ConstantDefaultWhat it bounds
agent.max_stepsresearch 10, code 60, computer 20, checker 8Per-agent cap on tool calls inside one subtask's agentic loop.
cycle threshold3 identical writes in a sliding windowIf the same write tool + args repeats 3×, the loop stops.
spawn_tool depth2One agent can spawn another, but not infinite delegation.
resolve re-routeboundedThe only path back into execution — fires solely on a genuine contradiction (γ' = γ + 0.2), capped.
recurrence_threshold2Similar buffered snippets required before RecMem spends an LLM to consolidate.
Read together: the worst case for a runaway is bounded by max_steps tool calls per subtask, with cycle detection killing anything that repeats and no automatic retry loop multiplying the work. There is no path to "ANet burned $200 overnight."
A worked example, revisited

The same request, with the data shown.

Same prompt as Level 1. This time we'll show the actual JSON the decomposer emits, the DAG metrics that make routing pick the sequential topology, the gate prompts that appear, and how synthesis composes the final reply.

You type into the terminal:

"Search the web for the top 5 most starred Python repositories on GitHub right now. Write a Python script that fetches their README (first 500 chars) via the GitHub API and saves a report.json with name, stars, and readme snippet for each. Write a test that validates the JSON structure and that all 5 entries exist. Run it, fix any failures, re-run until clean. Then send the final report.json contents as a formatted message to a Telegram bot."

Step 1The decomposer's JSON output.

The decomposer LLM runs once and emits the following — this is what Phase 1 returns and what dag.build() consumes: 

Decomposer output for this request{
  "trivial":       false,
  "request_class": "coding",
  "subtasks": [
    {
      "id":          "s1",
      "agent":       "research_agent",
      "description": "Find the top 5 most starred Python repositories on GitHub right now. Return name + star count for each.",
      "depends_on":  [],
      "complexity":  "low",
      "coupling":    "none"
    },
    {
      "id":          "s2",
      "agent":       "code_agent",
      "description": "Using the 5 repos from s1, write fetch_report.py (fetches each README via the GitHub API, takes the first 500 chars, saves report.json with name/stars/readme), write test_report.py that validates the JSON structure and all 5 entries exist, run pytest, fix any failures, re-run until all green.",
      "depends_on":  ["s1"],
      "complexity":  "high",
      "coupling":    "strong"
    },
    {
      "id":          "s3",
      "agent":       "checker_agent",
      "description": "Validate s2: confirm pytest passed and report.json is well-formed with all 5 entries.",
      "depends_on":  ["s2"],
      "complexity":  "low",
      "coupling":    "strong"
    },
    {
      "id":          "s4",
      "agent":       "tele_agent",
      "description": "Read report.json, format as a Markdown summary (one bullet per repo), send it to the configured Telegram chat.",
      "depends_on":  ["s3"],
      "complexity":  "low",
      "coupling":    "strong"
    }
  ]
}

Each depends_on names exactly one predecessor — s2→s1, s3→s2, s4→s3 — so dag.build() produces a single chain: ω = 1, |E| = 3. Note checker_agent is just subtask s3 here, not a separate stage. The coupling: "strong" edges (0.7 each) give a coupling density γ = 0.7, and the high-complexity code node dominates the critical-path depth δ.

Step 2Route reads the metrics.

Algorithm 1 runs on the built DAG. |E| = 3 (not zero), and ω = 1 — the very first rule after the empty-graph check fires: a single dependency chain routes to τ_S sequential. (The γ and δ values are computed but don't get a say here — ω = 1 decides it outright.) The sequential executor will run s1 → s2 → s3 → s4 in order, threading each subtask's output into the next.

Step 3What the gate actually says.

Inside code_agent's loop, every edit_tool and every shell_tool call hits _needs_confirm and stops to ask. In your terminal you'd see something like: 

Confirmation gate (terminal view)edit_tool wants to write fetch_report.py
  + 47 lines, - 0 lines  ·  full diff:
  + import requests, json
  + ...
[y] yes  [n] no  [a] all-for-this-step ›

shell_tool wants to run: pytest test_report.py -q
  cwd: /home/you/project
[y] yes  [n] no  [a] all-for-this-step ›

Press a on the first edit and you won't be asked again for that step — useful when you trust the agent on this run but not the next one. The "all" scope is one step, not one session.

Step 4Each subtask's output, one by one.

1

s1 runs research_agent

Result: a JSON-ish list of 5 repos with names and star counts. It stops calling tools, so its own loop ends and the output is threaded into s2.

2

s2 runs code_agent

Receives the repo list. Result after its internal loop: "report.json saved, pytest exited 0 (4 passed)." The iteration you see in the timeline is the agent's own loop — no outer stage graded it; it simply kept going until its tests were green (or it would have hit max_steps = 60).

Counter-factual: if the agent had exhausted max_steps with tests still red, its final output would say so honestly — and s3 (checker_agent) would report the discrepancy. There is no automatic re-run of the whole graph.

3

s3 runs checker_agent

A validation subtask, not a stage. Reads what s2 produced and verifies it: tests passed, report.json exists and is well-formed with 5 entries. Its verdict is just another subtask output that flows forward into synthesis.

4

s4 runs tele_agent · external

Result: "Sent — message_id 1283." Reads report.json, formats a Markdown summary, calls tele_tool.send_message(...). No subtasks remain.

Step 5Synthesis composes a reply.

Synthesis gets the four subtask outputs (each with agent, task, output) plus the topology. Because this was τ_S sequential, select_operator() returns sequential_last — the chain's last output already carries the earlier work forward, so it's emitted and framed rather than re-merged. Its prompt tells it to present the actual content, not just announce that work happened. Tokens stream live:

ANet replies in your terminal:

Done. Pulled the top 5 starred Python repos (public-apis/public-apis, vinta/awesome-python, donnemartin/system-design-primer, TheAlgorithms/Python, jackfrued/Python-100-Days). Wrote fetch_report.py and test_report.py. First test run failed (snippet length), fixed the slice to readme[:500] and re-ran — 4 tests passing. report.json saved at the project root, and the formatted summary was sent to your Telegram bot (message_id 1283).

Step 6What got persisted.

  • conversations.db — your request and ANet's reply, both stamped with this session's thread.
  • Rolling summary — unchanged this turn (window didn't overflow); will update later when it does.
  • RecMem — a raw snippet of this turn drops into the subconscious buffer (cheap embedding, no LLM). Nothing is consolidated unless a similar interaction recurs; if you asked for a repo report again later, that recurrence would trigger the LLM to distil a durable fact like "User tracks top starred Python repos and delivers reports via a Telegram bot."

Level 2 recap

The deep cuts, in five.

  1. Decompose emits one of two JSON shapes — a trivial reply or a set of subtasks, each with id, agent, description, depends_on, complexity/estimated_tokens, coupling.
  2. The DAG computes ω/δ/γ; Route runs Algorithm 1 — a deterministic rule table (no LLM) mapping the shape to τ_P / τ_S / τ_H / τ_X, logging every decision.
  3. The Orchestrator runs each subtask (unchanged). It dispatches to the agent's provider, intercepts every tool call, runs it through the per-tool/per-action confirmation policy, and feeds the result back. Bounded by max_steps and the cycle detector.
  4. There's no checker stage and no retry loop. Validation is a subtask (checker_agent) or the agent's own loop; synthesis (Algorithm 2) dispatches to one of five operators — compose / aggregate / vote / rank / resolve.
  5. State is small and shared. One conversations.db keyed by thread, one per-thread rolling summary, and RecMem — a 3-tier store that consolidates with the LLM only on recurrence.

That's the engine with its hood up.
