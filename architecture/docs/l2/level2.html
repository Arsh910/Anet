Stage 1 · Planner

What the Manager actually outputs.

The Planner is one model call, with one job: turn your natural-language message into a strictly-typed JSON object. Two shapes are valid — a simple reply, or a plan with a list of steps. Anything else is a bug. 

Planner output — schema// Shape A — direct reply (greetings, recall, trivial facts)
{
  "type":  "simple",
  "reply": "<assistant's answer>"
}

// Shape B — a DAG to execute
{
  "type": "plan",
  "steps": [
    {
      "id":                1,
      "agent":             "<agent name>",
      "task":              "<full instruction, self-contained>",
      "success_criteria":  "<verifiable outcome>",
      "check":             null,   // or {action, ...} for cheap deterministic checks
      "depends_on":        [],     // e.g. [1, 2] → wait for steps 1 and 2
      "wait_for_async":    false   // only matters for background async steps
    }
  ]
}

Every field is load-bearing. depends_on drives the executor's parallelism. success_criteria is what the Checker grades the result against. check (when not null) lets the engine skip the LLM checker for cheap deterministic cases — "is this window open?" / "does this file exist?".

Why JSON-only? The Planner prompt forbids prose. Even a follow-up like "explain what you just did" must come back as {"type":"simple","reply":"..."}. This makes the executor's input always parseable — no regex around free text, no "did the model drift today" branches.

Prompt anatomyWhat the Planner sees.

The system prompt is assembled fresh each turn from these pieces, in this order (see _plan_system_prompt in engine.py):

  1. Identity + date. "You are ANet. Current date: …" — so dated questions ("latest", "this week") have a frame.
  2. SOUL.md persona. The pack's personality file, injected verbatim if present.
  3. "What I know about you" profile. The 12 most relevant always-inject memories (preferences, identity).
  4. RELEVANT MEMORIES block. Semantic + keyword hits from mem0 against the current request.
  5. AVAILABLE AGENTS list. Each agent's name + its declared task_types — this is how routing actually works.
  6. The schema + rules. JSON spec, routing rules ("file_agent has no edit_tool"), correction-rule ("if the user pushes back, replan").
  7. Few-shot examples. 4 short pairs covering greeting, multi-agent, project scaffolding, correction.

Most of the prompt is fixed text, but those four memory/context blocks change every turn — which is how a standing preference like "always prefix functions with anet_" actually reaches a coding task even when you don't mention it.

Stage 2 · Executor

How "steps run in parallel" actually works.

The Executor is a small loop with three jobs: find every step whose dependencies are done, run all of those at once, then ask the Checker whether anything needs to retry. Repeat until no step is ready. 

engine.py — the loop, simplifiedwhile True:
    exec_r = await self._execute(plan, step_statuses, ...)   # asyncio.gather over ready steps

    if exec_r["last_step_count"] == 0:
        break   # nothing ran (all offloaded or no ready steps)

    check_r = await self._check(plan, exec_r["active_step_ids"], ...)

    if check_r and check_r.get("retry"):
        attempts = check_r["attempts"]
        continue   # same step, with the checker's adjustment

    attempts = 0
    if not _has_ready_steps(plan, step_statuses):
        break
Step statusMeaningWhat the loop does
pendingNot yet attempted.Eligible if all depends_on are success.
in_progressCurrently running inside gather.Skipped on the next ready-set scan.
successResult satisfied success_criteria.Unlocks any step that depends on it.
partialGot something but not enough.Retry up to _MAX_RETRIES = 3, with the checker's hint.
failureThe agent gave up or the check failed deterministically.Step is marked done (so the loop terminates); the Synthesizer reports it honestly.
Important nuance: a failure doesn't crash the plan. The Executor still finishes any independent branches, and the Synthesizer reports exactly which parts ran and which didn't. Partial success is a first-class outcome, not an exception.
Inside one step · the agentic loop

The Orchestrator — what one step really does.

Each step in the DAG is run by orchestrator.run(...). This is a tight loop: ask the agent's model what to do, intercept any tool call, check the safety policy, run the tool, feed the result back. It sounds simple — the details are where the safety lives.

2.1The confirmation policy table.

Not every tool call asks for approval — that would make ANet unusable. The orchestrator carries a small policy in _CONFIRM_TOOLS that says, per tool, exactly which actions need a y / n / a:

ToolWhich actions need approvalWhy
shell_toolevery commandArbitrary shell — always dangerous.
code_executionevery runRuns arbitrary Python.
edit_toolevery editModifies your files.
download_fileevery downloadWrites a file to disk.
file_toolwrite_file, create_folder, delete_file, copy_file, move_file, rename_file, zip_files, unzip_fileThe read actions (list_dir, read_file) pass through.
open_applaunch_and_type, type_text, click_element, keyboard_shortcutChanges desktop state. Plain "open" is silent.
memory_toolclear, deleteNever wipe memory silently.

Read-only operations are explicitly listed (_READ_ONLY_OPS, _READ_ONLY_ACTIONS) so the cycle detector knows to exempt them. The model can list a directory in a tight loop if it wants; it just can't write in one.

2.2Provider dispatch — agent_runner.py.

Every agent declares its own model and provider. agent_runner normalises that into a unified call. Two paths:

OpenAI-compatible

openrouter · openai · google · vertex_google

Uses AsyncOpenAI with a per-provider base_url. OpenRouter routes 300+ models. Gemini's "thought signature" extras are preserved verbatim in _message_to_dict so thinking models don't reject their own looped-back messages.

Anthropic native

anthropic · vertex_anthropic

Uses anthropic.AsyncAnthropic. Tool-call shape is translated to match the OpenAI shape the orchestrator already speaks — one common return type either way.

2.3Memory injection — every step, every time.

Before the model call, _memory_block(agent, task) builds a short "RELEVANT MEMORY" block by pulling:

  • Standing memories — always_inject categories scoped to this agent (your preferences, your identity). Surfaces even when there's no keyword overlap.
  • Task-relevant facts — semantic search against the task with min_score = 0.34, so "write a function" doesn't drag in every memory mentioning code.

Snippets are capped at 240 chars each — no single mega-memory can flood a small model's context. The block is empty (and omitted entirely) when nothing is relevant.

Stage 3 · Checker

How "good enough" gets decided.

The Checker runs after every batch of steps the Executor finishes. There are two flavours — a cheap deterministic one for things you can literally check (is this window open?), and an LLM one for everything else.

Explicit check

when step.check ≠ null

The Planner pre-declares "verify by opening the window 'Notepad'". The Checker just runs check_window / check_process / check_path — no LLM call. Fast and certain.

Implicit check (LLM)

when step.check is null

The Checker model reads the step's output, the original task, and the success_criteria, then emits one of three verdicts plus an optional adjustment string.

3.1The three verdicts.

VerdictTriggerEffect
success Output meets success_criteria. Step marked done. Dependent steps become eligible.
partial Output has something but missed a requirement. Checker writes an adjustment hint; Executor retries the step with that hint appended, attempts ++.
failure Output is empty / wrong tool / errored. Step is closed; Synthesizer reports it honestly. Dependent steps are skipped.

3.2The retry budget.

engine.py — line 24_MAX_RETRIES = 3

# Inside the check phase:
advance = status == "success" or attempts >= _MAX_RETRIES - 1

Three attempts total, ever. After the third partial, the step advances anyway — the Synthesizer will report what got done, the user can decide to ask for more. This is the load-bearing rule that keeps the system from looping on its own work for an hour.

Stage 4 · Synthesizer

One coherent reply, streamed.

With every step labelled, the Synthesizer composes the final user-facing reply. It's the only stage that writes plain prose (every other stage speaks JSON). The output is streamed token-by-token to the terminal.

  1. Input: the full list of step_results (agent name, task, raw output, status) plus the original user message.
  2. System prompt: "present the actual content fully, don't summarise it away, and if any step partially ran, say exactly what was completed and what was skipped."
  3. Streaming: tokens arrive over the network and are printed live — no perceived "stuck" gap between the last agent finishing and the user seeing a reply.
  4. Dedup pass (_dedup_reply): removes accidental repeated paragraphs some models emit on long syntheses. Defensive, rarely triggers.
  5. Persistence: reply is stored in conversations.db under the thread, ready for next turn / --resume.
Where state lives

One DB, one summary, one vector store.

Three things outlive a single request. Knowing where each lives makes session switching, --resume, and memory recall stop feeling like magic.

conversations.db

sessions/conversations.db

One shared SQLite file, accessed via aiosqlite. Rows are keyed by thread — a session is just a thread name. Switching /session is a string change, not a DB swap.

Rolling summary

in-engine + same DB

Each thread has a single text summary covering everything older than the recent verbatim window. Recomputed (one LLM call) only when the window overflows recent_tokens — typically once every several turns, not every turn.

mem0 + Chroma

~/.anet/memory/

Long-term facts. mem0 reads the recent conversation every incremental_interval = 5 turns (and once on clean exit), extracts durable facts, de-duplicates, and writes to a local Chroma DB with on-device fastembed embeddings.

engine.py — when extraction firesif interval > 0 and n % interval == 0:
    async def _extract(snap=messages_snap, tid=thread_id):
        await asyncio.to_thread(memory_store.add_conversation, snap[-30:], run_id=tid)
    asyncio.create_task(_extract())   # fire-and-forget; turn returns immediately

The extraction runs in a background thread (mem0 is sync) so the user's next prompt isn't blocked by an LLM call they didn't ask for. If extraction fails or the model is unreachable, the turn just doesn't get extracted — no error reaches the user.

The constants that bound everything

Five numbers that hold the line.

ANet's safety isn't policy text — it's hard caps in code, low enough that a runaway loop hits one of them in seconds, not minutes.

ConstantDefaultWhat it bounds
_MAX_RETRIES3How many times a single step can come back partial before the engine moves on.
agent.max_stepsresearch 10, code 60, file 25, computer 20, checker 8Per-agent cap on tool calls inside one step's agentic loop.
cycle threshold3 identical writes in a sliding windowIf the same write tool + args repeats 3×, the loop stops.
spawn_tool depth2One agent can spawn another, but not infinite delegation.
incremental_interval5Turns between mem0 fact-extraction passes.
Read together: the worst case for a runaway is bounded by max_steps × _MAX_RETRIES tool calls per step, with cycle detection killing anything that repeats. There is no path to "ANet burned $200 overnight."
A worked example, revisited

The same request, with the data shown.

Same prompt as Level 1. This time we'll show the actual JSON the Planner emits, the gate prompts that appear, and the Checker's verdict at each hand-off.

You type into the terminal:

"Search the web for the top 5 most starred Python repositories on GitHub right now. Write a Python script that fetches their README (first 500 chars) via the GitHub API and saves a report.json with name, stars, and readme snippet for each. Write a test that validates the JSON structure and that all 5 entries exist. Run it, fix any failures, re-run until clean. Then send the final report.json contents as a formatted message to a Telegram bot."

Inside code_agent's loop, every edit_tool and every shell_tool call hits _needs_confirm and stops to ask. In your terminal you'd see something like:

Press a on the first edit and you won't be asked again for that step — useful when you trust the agent on this run but not the next one. The "all" scope is one step, not one session.

Level 2 recap

The deep cuts, in five.

  1. The Planner emits one of two JSON shapes — simple or plan. Each step is a 7-field record: id, agent, task, success_criteria, check, depends_on, wait_for_async.
  2. The Executor is a ready-set + asyncio.gather loop. Independent steps run concurrently; dependents wait. Retries cap at _MAX_RETRIES = 3.
  3. The Orchestrator runs each step. It dispatches to the agent's provider, intercepts every tool call, runs it through the per-tool/per-action confirmation policy, and feeds the result back. Bounded by max_steps and the cycle detector.
  4. The Checker has two flavours: deterministic (check_window/check_path/check_process) when the Planner pre-declared one, LLM-judged otherwise. Three verdicts: success / partial / failure.
  5. State is small and shared. One conversations.db keyed by thread, one per-thread rolling summary, one local mem0 store that extracts facts every 5 turns.

That's the engine with its hood up.
