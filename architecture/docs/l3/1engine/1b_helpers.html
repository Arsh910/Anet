_profile_context lines 43–67

engine.py:43–67def _profile_context(limit: int = 12) -> str:    try:        from anet.core import memory_store        if not memory_store.is_available():            return ""        mems = memory_store.get_all(limit=200)        if not mems:            return ""        # Standing memories first (identity/preferences), then recent facts.        prefs = [m for m in mems if m.get("always_inject")]        facts = [m for m in mems if m not in prefs]        facts.sort(key=lambda m: m.get("created_at", ""), reverse=True)        picked, seen = [], set()        for m in prefs + facts:            c = (m.get("content") or "").strip()            if c and c not in seen:                seen.add(c)                picked.append(c)            if len(picked) >= limit:                break        if not picked:            return ""        return "\n".join(f"- {c}" for c in picked)    except Exception:        return ""
What it produces: a markdown-ish bullet list of up to 12 user facts, prepended into the planner prompt under "What I know about you". This is how the planner already knows your name, your role, and your standing preferences before you type anything.
Selection order matters. Standing memories (always_inject categories: identity + preference) come first because the planner needs them to apply per-turn — they don't have to be relevant to the current message. Then plain facts come in by recency. The seen set de-duplicates by content so two memories carrying the same fact don't both win a slot.
limit=12 is a deliberate small number. The planner prompt is already long (see chapter 1c); 12 facts at ~80 chars each adds ~1 KB. Going higher dilutes the model's attention on the actual task. The cap is a per-function default; the engine never overrides it.
The outer try/except is the same defensive pattern as _load_soul_once: if memory is unavailable (no key, broken Chroma, fresh install), this function returns "" and the planner prompt just omits the profile section. Memory failures are never allowed to crash a turn.

The two knobs that govern short-term memory:
  • recent_tokens = 3000 — the engine will keep enough recent turns verbatim to fit within this many tokens; anything older gets summarised into the rolling summary.
  • min_recent = 4 — even if a single message blows the 3000-token budget, the last 4 messages are kept verbatim no matter what. This stops the summariser from "eating" the in-progress conversation.
Both are overridable per pack via context: in anet.config.yaml; you'll see them again in chapter 1e when we walk _maintain_summary.

The flow:
  1. Resolve (model, provider) from the pack config.
  2. If the provider is one of the three Vertex variants, delegate to agent_runner.build_vertex_client() which uses Application Default Credentials — different auth, different return path (covered in chapter 03).
  3. Otherwise look up (base_url, env_key) in _MANAGER_PROVIDERS.
  4. Read the env var. If missing, raise RuntimeError with the exact env-var name you need to set. This is allowed to bubble up — unlike SOUL/profile, you literally cannot plan without an API key.

Unknown provider behaviour: the .get(provider, _MANAGER_PROVIDERS["google"]) silently falls back to Google if the pack config names a provider that isn't in the table. That's chosen on purpose — the alternative (raise on unknown) makes upgrading a typo'd config destructive. Falling back with a missing-key error is recoverable.

_extract_json — the tolerant parser lines 118–136

engine.py:118–136def _extract_json(text: str) -> dict:    text = text.strip()    try:        return json.loads(text)    except json.JSONDecodeError:        pass    m = re.search(r"```(?:json)?\s*([\s\S]+?)\s*```", text)    if m:        try:            return json.loads(m.group(1))        except json.JSONDecodeError:            pass    m = re.search(r"\{[\s\S]+\}", text)    if m:        try:            return json.loads(m.group(0))        except json.JSONDecodeError:            pass    raise ValueError(f"No JSON found in: {text!r}")
Three escalating attempts to recover JSON from a model reply:
  1. Strict parse — the planner prompt says "OUTPUT ONLY JSON", and 99% of the time that's exactly what comes back.
  2. Fenced parse — some models can't help wrapping JSON in ```json ... ```. Strip the fence and try again.
  3. Greedy regex parse — find the first { and the last } and try the slice between them. Catches "Here's the plan: {…}" cases.
Only after all three fail do we raise. The caller (_plan in chapter 1e) catches the ValueError and falls back to _keyword_fallback — see below.
Why not use a JSON-mode API? Because the engine has to work across providers — OpenRouter, Vertex, openai-compatible endpoints — and not all of them support JSON-mode reliably for every model. A tolerant parser is more portable than a feature flag.

_dedup_reply lines 158–166

engine.py:158–166def _dedup_reply(text: str) -> str:    n = len(text)    if n < 100:        return text    start = text[:80]    idx = text.find(start, n // 3)    if idx > 0:        return text[:idx].rstrip()    return text
The "did the model paste its answer twice?" guard. Some thinking models occasionally re-emit their full answer at the end of a streamed response. This function takes the first 80 characters of the reply, searches for that same prefix anywhere after the first third of the text, and truncates at the match if found. Defensive and cheap; almost never triggers.
Why n // 3 and not just 1? Because a reply that legitimately begins "ANet replies: …" and happens to repeat the phrase "ANet replies:" partway through (e.g. inside a code example) would otherwise be falsely truncated. Starting the search at the one-third mark makes false positives essentially impossible.

engine.py:169–185def _keyword_fallback(user_msg: str, agents: list[dict]) -> dict:    plannable = [a for a in agents if a["name"] != "checker_agent"]    lower = user_msg.lower()    best, best_score = plannable[0], 0    for a in plannable:        score = sum(            1 for tt in a.get("task_types", [])            for w in tt.lower().split() if w in lower        )        if score > best_score:            best_score, best = score, a    print(f"[engine] keyword fallback → '{best['name']}'", file=sys.stderr)    return {"type": "plan", "steps": [{        "id": 1, "agent": best["name"], "task": user_msg,        "success_criteria": "Task completed without errors.", "check": None,        "depends_on": [], "wait_for_async": False,    }]}
The last-resort planner. If the LLM planner returns something _extract_json can't salvage, the engine still has to do something. This function scores each agent's task_types words against the user message and routes the whole request as a one-step plan to the best match. The criterion is naive (string-in-string), the result is a plausible single-step plan — better than crashing.
Excludes the checker from candidates because the checker isn't a worker agent; planning a task to the checker makes no sense.
The print(..., file=sys.stderr) is the one logging call in this function. We want this rare event to be visible to operators (it usually signals a flaky free model returning garbage) without disrupting the TUI, which only writes to stdout.

_has_ready_steps — the executor's continue test lines 188–205

engine.py:188–205def _has_ready_steps(plan: list[dict], step_statuses: dict) -> bool:    for step in plan:        sid = str(step.get("id", step.get("agent", "")))        if step_statuses.get(sid) in ("completed", "offloaded", "failed"):            continue        deps       = step.get("depends_on", [])        wait_async = step.get("wait_for_async", False)        blocked = any(            step_statuses.get(str(d), "pending") == "offloaded" and wait_async            for d in deps        )        pending_deps = any(            step_statuses.get(str(d), "pending") in ("pending", "running")            for d in deps        )        if not blocked and not pending_deps:            return True    return False
Returns True the moment it finds any step that can run next. Used by the executor loop in run_turn as the condition to keep iterating. The function:
  1. Skips steps that are already completed, offloaded (running async in background), or failed.
  2. For each remaining step, looks at depends_on.
  3. If any dep is offloaded and this step is marked wait_for_async, the step is blocked (waiting for the background result).
  4. If any dep is still pending or running, the step isn't ready yet.
  5. Otherwise → True: this step is ready to run.
Short-circuits. Returns at the first ready step — no need to enumerate all of them. The executor will scan again on the next iteration anyway. The function is called many times per turn, so early-out matters.
The "offloaded but not awaited" case. If a step depends on an offloaded step but doesn't set wait_for_async, it's treated as ready — the dependent step runs without waiting for the background task. This is what enables fire-and-forget patterns (e.g. start a 3D render in background, immediately reply to the user, notify when the render completes).

The "no LLM needed" checker. When the planner pre-declared a check (e.g. {"action": "check_window", "title": "Notepad"}), the Engine doesn't call the checker LLM — it calls this function, which calls the checker tool, which actually looks at your OS to verify.
Three deterministic actions:
  • check_window — is a window with this title open? (used after computer_agent opens an app)
  • check_process — is a named process running?
  • check_path — does this file or folder exist?
Each returns ("success", description) or ("failure", description). The string is presented to the user verbatim in the Synthesizer's input.
The notify callback is the on_status context-var from chapter 1a, passed in by the caller. It writes "checker: check_window(title='Notepad')…" to the status line of the TUI, so the user sees the check happening live.
Unknown action returns "unknown" — not "failure". The Engine's _check caller falls back to the LLM checker on "unknown", so a typo in the action name degrades gracefully to LLM-judged checking instead of hard-failing.
