the engine · section 2 of 3

The shared and phase helpers.

Two groups. The shared helpers on BaseEngine and engine_base.py — rolling summary, manager client, persistence — that every engine inherits. And the phase helpers scattered across the AdaptOrch package: the context renderer, the DAG metrics, the router decision, the executors' context threading, and the per-stage model calls. Anything the pipeline in chapter 1a leaves un-explained, one of these already did it.

Part A — shared helpers (engine_base.py)

_maintain_summary — short-term memory engine_base.py:113–159

engine_base.py:113–159 · excerptasync def _maintain_summary(self, store, thread_id, messages) -> tuple[str, int]:    from anet.core import context_window as cw    recent_tokens, min_recent = _context_settings()    summary, count = await store.get_summary(thread_id)    count = max(0, min(count, len(messages)))    keep_from, overflow = cw.plan_window(messages, recent_tokens, min_recent, count)    if overflow:        client, model = _manager_client()        msgs = cw.build_summary_messages(summary, overflow)        resp = await client.chat.completions.create(            model=model, temperature=0, max_tokens=700, messages=msgs)        new_summary = (resp.choices[0].message.content or "").strip()        if new_summary:            summary = new_summary            await store.set_summary(thread_id, summary, keep_from)    self._summaries[thread_id] = summary    return summary, keep_from
The rolling summary + token-budgeted window. This is the whole short-term-memory strategy. It hands the pure planner in context_window.py (chapter 12) the messages, the token budget, and how many turns are already summarised; plan_window returns a keep_from cut and the overflow — the turns that have just scrolled out of the verbatim window. Everything at messages[keep_from:] is sent verbatim; everything before is covered by summary.
Summarisation fires only on overflow — typically once every several turns, not every turn. When it does, it's one manager call (temperature=0, max_tokens=700) that folds the overflow into the running summary, then persists it via store.set_summary. If nothing overflowed, there's no LLM call at all. The real file also caches the fixed summariser system prompt when the provider supports Anthropic-style prompt caching.
Failure is safe. If the summariser call raises, the code sets keep_from = count — it refuses to advance past what's already summarised, so no turn is ever dropped from view without coverage (the window may temporarily exceed budget instead). Memory maintenance never loses a turn.

The one write per turn. Appends the user message (unless this is an async resume, where the user message was already stored) and the assistant reply to the conversation store. It's a @staticmethod because it needs no engine state — it's pure store plumbing. Every exit path in run_turn (trivial fast-path, normal, and the error guard) routes through it, so the conversation is always consistent.

The manager client — used for the summary call and as the default per-stage model. _manager_cfg resolves (model, provider) from the pack's manager: config, defaulting to gemini-2.5-pro on Google. _manager_client builds the client: the three OpenAI-compatible providers use a base_url + env-var lookup; the Vertex variants delegate to agent_runner.build_vertex_client() (Application Default Credentials). A missing API key raises with the exact env-var name — you literally can't run without it.
Where the manager model is used: directly for _maintain_summary's summariser call. It's also the fallback for every orchestration stage — the decomposer, merger, and arbiter all resolve to the manager model unless a per-stage override is set. See stage_models in Part B.
Unknown provider → Google. The .get(provider, _MANAGER_PROVIDERS["google"]) falls back silently if the config names a provider that isn't in the table. Chosen on purpose: the alternative (raise on unknown) makes a typo'd config destructive; a fallback with a missing-key error is recoverable.

_context_settings engine_base.py:54–62

engine_base.py:54–62def _context_settings() -> tuple[int, int]:    """(recent_tokens, min_recent) for the short-term window."""    try:        from anet.core.config_loader import load        cfg = load().get("context", {}) or {}        return int(cfg.get("recent_tokens", 3000)), int(cfg.get("min_recent", 4))    except Exception:        return 3000, 4
The two knobs that govern short-term memory:
  • recent_tokens = 3000 — keep enough recent turns verbatim to fit within this many tokens; anything older folds into the rolling summary.
  • min_recent = 4 — even if a single message blows the budget, the last 4 messages are kept verbatim. This stops the summariser from "eating" the in-progress exchange.
Both are overridable per pack via context: in anet.config.yaml, and both feed straight into _maintain_summary above.

Same context as the chat window, flattened to one string. DAG metrics — TaskDAG.features / summary dag/__init__.py:87–103

dag/__init__.py:87–103 · excerptdef features(self) -> dict:    return {        "num_nodes": self.num_nodes, "num_edges": self.num_edges,        "omega": self.omega, "delta": round(self.delta, 4),        "gamma": round(self.gamma, 4), "parallelism_ratio": round(self.parallelism_ratio, 4),        "num_layers": len(self.layers),    }def summary(self) -> str:  # one-line shape for the status line    return f"{self.num_nodes} subtasks · {len(self.layers)} layers · ω={self.omega} δ={self.delta:g} γ={self.gamma:.2f}"
The structural metrics Phase 2 computes, in one place. build() (chapter for the DAG file) runs Kahn layering and one DP pass to fill in:
  • ω (omega) — parallelism width, the widest layer: how many subtasks can run at once.
  • δ (delta) — weighted critical-path depth: the longest dependency chain by cost weight.
  • γ (gamma) — coupling density, Σc / |E|: the average context-sharing across edges.
  • layers — the topological layering the hybrid executor walks.
features() is the flat dict the router logs; summary() is the human one-liner you see on the status line.

The router decision — Algorithm 1 router/__init__.py:80–113

router/__init__.py:80–113 · excerpt_DEFAULT_THRESHOLDS = {"theta_omega": 0.5, "theta_gamma": 0.6, "theta_delta": 5.0}def route(dag, *, th=None, gamma_override=None) -> RouteDecision:    t_omega, t_gamma, t_delta = th["theta_omega"], th["theta_gamma"], th["theta_delta"]    if E == 0:                                topo = Topology.PARALLEL      # τ_P — no deps    elif omega == 1:                         topo = Topology.SEQUENTIAL    # τ_S — single chain    elif gamma > t_gamma and V > t_delta:    topo = Topology.HIERARCHICAL  # τ_H — coupled + big    elif r > t_omega and gamma <= t_gamma:   topo = Topology.PARALLEL      # τ_P — wide, low coupling    else:                                    topo = Topology.HYBRID       # τ_X — mixed
Pure, deterministic, O(1). The router reads the metrics the DAG already computed (ω, γ, |V|, |E|, and r = ω/|V|) and applies the five-rule Algorithm 1 against the calibrated thresholds θ_ω=0.5, θ_γ=0.6, θ_δ=5 — overridable per pack via the orchestration: block. No LLM. The gamma_override argument lets synthesis re-route with an inflated coupling estimate (γ' = γ + 0.2) without rebuilding the DAG. Every call returns a RouteDecision carrying the topology, the human-readable reason (with the numbers), and a features snapshot for the log.

Executor context threading executors/__init__.py:55–94

The executors are pure topology + context assembly. They never call an LLM or touch agents — they schedule subtasks and build each one's prompt, then call the injected ctx.run_subtask (wired to orchestrator.run in chapter 1a). compose assembles the prompt every subtask receives: Shared context → Relevant prior results → Your task. accumulate_predecessors is the relevance-weighted truncation — it ranks a subtask's predecessor outputs by coupling strength (most-coupled first) and fits them into a char budget.
Each topology threads context differently. τ_P gives each subtask global context only (they're independent); τ_S accumulates predecessor outputs down the chain; τ_H has a lead delegate with isolated sub-agents; τ_X rolls prior layers forward. execute(topology, dag, ctx) dispatches to the right one via get_executor.

stage_call / stage_call_stream stage_models.py:26–121

stage_models.py:26–84 · excerptdef stage_model(stage: str) -> tuple[str, str]:    # orchestration.<stage>.{model,provider} → manager.{model,provider} → defaults    stage_cfg = (cfg.get("orchestration") or {}).get(stage) or {}    mgr = cfg.get("manager") or {}    return (stage_cfg.get("model") or mgr.get("model") or _DEFAULT_MODEL,            stage_cfg.get("provider") or mgr.get("provider") or _DEFAULT_PROVIDER_NAME)async def stage_call(stage, messages, *, max_tokens=1500, temperature=0.0) -> str:    client, model = build_stage_client(stage)    resp = await client.chat.completions.create(model=model, messages=messages, ...)    _tok.record(resp, stage=stage)          # per-stage token accounting    return (resp.choices[0].message.content or "").strip()
How every LLM-using phase resolves its model. The decomposer and synthesizer call stage_call / stage_call_stream with a stage name ("decomposer", "merger", "arbiter"). stage_model resolves the model in a strict order: a per-stage override under orchestration: → the manager model → built-in defaults. So by default everything runs on the manager model, and you override a stage only when you want to (e.g. a cheap model for decomposition, a stronger one for synthesis).
stage_call_stream is the streaming variant — it passes each content delta to the on_token sink as it arrives, so the synthesizer's final answer types out live in your terminal. Both variants record token usage tagged by stage (via tokens.py), which is what feeds the per-phase usage in the routing log.
Note what these do not do: there's no per-stage orchestration.mode flag and no legacy fallback. The five phases always run; the only per-stage knob is the model.
