The real prompts.

The pipeline makes exactly two kinds of LLM call that need a carefully-written prompt: the decomposer (Phase 1) that turns your request into a subtask DAG, and the synthesizer (Phase 5) that merges the results. The decomposer has one system prompt; the synthesizer has one per operator. These prompts are the behaviour spec for how ANet plans and reports — change a rule and the whole assistant changes.

Why prompts deserve their own chapter: everything else in the pipeline is deterministic Python (the DAG metrics, Algorithm 1's routing, the executors' scheduling). The only places a model gets free rein are these two, so this is where the actual "intelligence" of the plan and the final answer is shaped. Read them carefully.

decomposer._SYSTEM — the decomposer's brain decomposer/__init__.py:43–59

Based on the AdaptOrch paper's §B.1 decomposition template, extended with agent routing. It never executes — it only plans. 

decomposer/__init__.py:43–59 · the system promptYou decompose a user request into a DAG of subtasks and route each to an agent.You never execute — only plan.FAST-PATH: for trivial requests (greeting, single fact, one-liner) return`{"trivial": true, "reply": "<answer>"}` and stop.Otherwise emit the fewest subtasks that capture real structure.Over-decomposition wastes tokens.RULES:- MAXIMIZE PARALLELISM: add a depends_on entry ONLY when one subtask's output  is genuinely an input to another. Anything else stays `[]` so it can run concurrently.- COUPLING vs a dependency: strong=its output is your direct input; weak=shared  domain only; critical=semantic coherence required across both; none=independent.- TOOL-COMPLETE ROUTING: route each subtask to the SINGLE agent that can finish  it end-to-end with its own tools. Don't split across agents to borrow a tool.- Typical shapes: coding chains (localize files → patch → verify); research fans  out then in. Adapt to the actual task.
Four load-bearing rules, each shaping the DAG the router later reads:
  • FAST-PATH — the trivial short-circuit. A greeting or one-liner returns {"trivial": true, "reply": ...} and the whole pipeline stops here (chapter 1a). No agents, no synthesis.
  • MAXIMIZE PARALLELISM — dependencies are added only when one output truly feeds another. This directly controls ω (width) and δ (depth): sparse dependencies → wide DAG → the router picks a parallel topology.
  • COUPLING — the four labels (none / weak / strong / critical) map to the numeric c(u,v) ∈ [0,1] the DAG uses to compute γ. This is how the model's judgement about "how tied together are these steps" flows into Algorithm 1.
  • TOOL-COMPLETE ROUTING — each subtask goes to the single agent that can finish it end-to-end, so the executor never has to hand a half-done task between agents to borrow a tool.
The DAG vocabulary is the paper's, made concrete. The prompt teaches the model the exact annotations the pipeline consumes — depends_on, coupling, complexity/token estimates — because those are what dag.build() turns into ω / δ / γ. The prompt and the pure DAG code are two halves of the same contract.

The output schema decomposer/__init__.py:61–87

decomposer/__init__.py:61–87 · the template + output spec (excerpt)Analyze the following task and decompose it into independent subtasks.For each: id+description, dependencies, Estimated complexity(tokens: low/medium/high) or estimated_tokens, Context coupling withdependencies (none/weak/strong/critical), agent.Task: {T}Available agents:{agents}Return ONLY this JSON (no prose):{  "trivial": false,  "request_class": "coding" | "reasoning" | "rag" | "general",  "subtasks": [{    "id": "s1", "description": "<what to do>", "agent": "<from list above>",    "depends_on": [], "complexity": "low" | "medium" | "high",    "estimated_tokens": 500, "coupling": "none" | "weak" | "strong" | "critical"  }]}
Every field maps to something the pipeline uses:
  • request_class (coding / reasoning / rag / general) — feeds the synthesizer's operator selection. A rag class defaults synthesis to aggregate; everything else to compose.
  • depends_on — the DAG edges. [] means "run me concurrently."
  • estimated_tokens or complexity — the cost weight w_i. A concrete token count is preferred; otherwise the coarse bucket {low:1, medium:3, high:9} is used. These weights drive δ (the weighted critical path).
  • coupling — one label for all deps, or a per-dep object {"<dep_id>": "strong"}. Missing labels fall back to the "strong" default (an edge exists because u feeds v).
  • agent — chosen from the injected {agents} list, which is formatted from each agent's task_types. A bad name falls back safely (chapter 1a's _resolve_agent).
The context is prepended, not embedded. When the coordinator passes memory_ctx (the rolling summary + recent turns from chapter 1a), build_prompt prepends it as a "Relevant context from memory:" block before the task. That's what lets the decomposer resolve follow-ups against prior turns.
Output is parsed tolerantly. The decomposer's reply goes through extract_json (in stage_models.py): strict parse, then a fenced-code parse, then a greedy {...} slice. Only if all three fail does it raise — caught by the coordinator's outer guard as a clean error. There's no separate keyword-fallback planner any more.

synthesizer — the four operator prompts synthesizer/__init__.py:85–102

Phase 5 is a dispatcher over five operators (compose / aggregate / vote / rank / resolve). Four of them are terminal LLM calls, each with its own tight system prompt; vote is embedding-based and needs no prompt. The operator is chosen by select_operator from the topology and request class — vote / rank / resolve are never chosen by default, only via an explicit hint or a declared-redundant decomposition. 

synthesizer/__init__.py:85–102 · the operator system prompts_COMPOSE_SYS = (    "Each output below is a DIFFERENT part of one task (e.g. backend, frontend, tests). "    "Integrate every part into one coherent deliverable — keep every part, don't compare "    "or pick. Return the final result.")_AGGREGATE_SYS = (    "Synthesize the findings below into one unified summary. Preserve every distinct piece "    "of information. Rank them by quality and return "    "the single best one (briefly say why). Don't merge — choose.")_RESOLVE_SYS = (    "The outputs make CONTRADICTORY claims on the same point. Decide which is right "    "(prefer freshly-retrieved evidence over prior knowledge), reconcile, and return "    "the correct answer.")
Each prompt asks the right question for its situation:
  • compose — the default for distinct, complementary parts (backend / frontend / tests). "Keep every part, don't compare or pick." This is what runs for a normal multi-agent task like our running example.
  • aggregate — the default for research / RAG. Unify findings, preserve every distinct fact, note disagreements rather than silently picking.
  • rank — for candidate generation (N options for one goal). Choose the best, don't merge. Opt-in only.
  • resolve — for a genuine contradiction. Prefer freshly-retrieved evidence over prior knowledge. This is the only operator that can trigger a re-route (below).
Two operators aren't LLM prompts at all. single and sequential_last are pure passthroughs — a one-output result, or the last link of a sequential chain, is already a coherent answer, so it's emitted directly. vote is embedding-based: it picks the output most similar to the rest as the consensus. All four terminal prompts stream token-by-token via stage_call_stream (chapter 1b), which is why you see the final answer typing out live.

The resolve loop — Algorithm 2's re-route synthesizer/__init__.py:197–211

synthesizer/__init__.py:197–211 · excerpt# Resolve — true contradiction; arbitrate, with the reroute loop (Algorithm 2).iters, bump = 0, 0.0while True:    iters += 1    out = await _combine("arbiter", _RESOLVE_SYS, outs, task)    post = alignment(out, outs, embed)    if (topology == Topology.HIERARCHICAL or post >= theta            or reroute_fn is None or iters >= max_iters):        return SynthesisResult(out, "resolve", topology, cs, iters, iters > 1, history)    bump += 0.2    topology, cur_results = await reroute_fn(base_gamma + bump)
The one place the pipeline loops back. If resolve arbitrates a contradiction and the result still doesn't align, it calls the reroute_fn the coordinator handed it (chapter 1a) with an inflated coupling estimate (γ' += 0.2), which re-routes and re-executes. It's bounded by max_iters and short-circuits for hierarchical topologies. Crucially, this is the only feedback loop — there's no planner/executor/checker retry loop, no success/partial/fail bounce-back.
Consistency score is a signal, not a gate. The paper's Algorithm 2 gated merge-vs-arbitrate on a global consistency score, which misfired on parallel subtasks that are distinct by design (three different facts look dissimilar → spurious arbiter). Here CS is demoted: it's only consulted inside vote and resolve. The normal distinct-subtask case defaults to compose/aggregate and never spuriously arbitrates.

The summariser prompt — a shared aside context_window.py:132–140

context_window.py:132–140 · _SUMMARY_SYSTEMYou maintain a running summary of a conversation between a user and an AIassistant. Given the summary so far and the new messages that are about toscroll out of the live window, return an updated summary that PRESERVES everydurable fact: decisions made, file paths, names, numbers, the user's goals andconstraints, and unresolved threads. Drop pleasantries and superseded details.Keep it tight and factual — bullet points or short sentences. Return ONLY theupdated summary text.
The third prompt in the engine's orbit. Not a phase, but the manager call that _maintain_summary (chapter 1b) fires when the recent-token window overflows. It's shared across engines because it belongs to short-term memory, not to any one pipeline. It preserves durable facts (paths, names, decisions, unresolved threads) and drops pleasantries.

What this section leaves you with

Every prompt the engine sends, in one place:

  • Decomposer (_SYSTEM + template + output spec) → your request becomes a DAG of subtasks with dependencies, coupling, cost weights, and per-subtask agent routing.
  • Synthesizer operators (_COMPOSE_SYS / _AGGREGATE_SYS / _RANK_SYS / _RESOLVE_SYS) → the results become one streamed answer, via the operator that fits; vote and the passthroughs use no prompt.
  • Summariser (_SUMMARY_SYSTEM) → the shared short-term-memory rolling summary.
