The two system prompts.

Lines 240–453. Two functions that build the only two prompts engine.py ever sends — the planner's instructions and the synthesizer's instructions. Most of the bytes are static text, but the dynamic pieces (agent list, profile, memory) are what make ANet adapt to your setup.

Why prompts deserve their own chapter: these two functions are the behaviour specification for ANet's planning and reporting. Change a rule here and the whole assistant changes. Changing them carelessly is the fastest way to regress the system. Read them carefully.

What's assembled, and from where:
  • plannable — every agent except the checker. The checker is a worker the engine uses internally; routing the user's request to it would be a category error.
  • agent_lines — the visible directory: "- research_agent: web research, news, image downloads". The Planner reads this verbatim and routes by matching task_types.
  • agent_names — the allowed values for the "agent" field in the planner's output JSON. Listed in the prompt so the model can't invent an agent that doesn't exist.
  • now — the current date and time, formatted for a human ("Sunday, June 21, 2026 16:43"). Letting the model know what "today" means is what unlocks "find the latest X" queries.
  • memory_section — the task-relevant memories from _memory_context (chapter 1b), passed in from _plan.
  • soul_section — the pack's persona, loaded once at import.
  • profile_section — the standing user facts from _profile_context.
Each section is conditional — the empty string collapses to nothing in the f-string interpolation. A pack without SOUL.md, with no memories yet, just yields a shorter prompt. Nothing breaks; the planner is just less personalised.

"Always JSON" is the single most important rule. The Engine's _extract_json can recover from a fenced reply, but the model returning "Sure, I'll do that!" in plain prose crashes the entire turn. The rule is restated several times in this prompt for that reason. 
engine.py:319–324 · the correction rule- CORRECTION RULE (HIGHEST PRIORITY — overrides everything else): If the user indicates the  previous result was wrong, incomplete, or not what they wanted — ANY phrasing like "you didn't  do X", "you didn't make X", "that's not what I asked", "you only did Y not Z", "you missed X",  "do it properly", "complete it", "try again", "still not working", "I asked for X" — you MUST  generate a new plan and dispatch to the appropriate agent. The correction rule fixes a specific failure mode. Without it, a Gemini-class model would respond to "that's not what I asked" with a type:"simple" apology — and the user is left exactly where they were. The rule forces a re-plan instead. Every "phrasing like" example came from real conversations where the previous prompt drifted into apologise-and-stop behaviour.

The "code_agent must not absorb" rule is what makes the Telegram step in our running example get its own step, routed to tele_agent, instead of letting code_agent use requests to call the Telegram API itself. Specialised agents hold credentials and conventions that a general agent shouldn't have to know about.
The CRITICAL blocks address specific historical confusions:
  • computer_agent vs code_agent — early planners would open a Terminal window via computer_agent and try to type into it. Wasteful and unreliable; code_agent has shell_tool already.
  • file_agent vs code_agent — file_agent has no edit_tool, so routing "fix this file" to it always failed. The fallback ("ALWAYS choose code_agent in doubt") is the safe answer.

Domain-specific rules lines 360–389

Three blocks for behaviours that aren't general planning — they're domain knowledge the planner needs to do its job correctly:

BlockWhat it forcesWhy it's here
APPROVAL GATE — 3D rendering If user asks for a 3D model without an image, plan ONLY the image download. Stop. Don't chain the render in the same plan. 3D rendering takes minutes and uses GPU. Approving the image first prevents a wrong-image render.
PATH INJECTION — viga_agent Scan assistant messages for "Downloaded: …" lines and put that path verbatim into the viga task. So viga_agent doesn't have to ask the user "which image?" — it gets the path injected.
IMAGE DOWNLOAD TASKS Tell research_agent to find a DIRECT image URL ending in .jpg / .png / .webp. Use Wikimedia Commons first. Without this, the research agent would return article links that aren't downloadable.
TELEGRAM TASKS Always set success_criteria to "Result must contain 'message_id' confirming Telegram delivery" So the checker has a verifiable token to grep for, not just the agent's claim "I sent it".
These blocks accrete. The planner prompt grows by bullet every time a recurring failure mode is identified. They're kept in this one prompt (rather than scattered across agents) because the planner is the only thing that can prevent a wrong delegation in the first place.

The synthesizer prompt has one job: turn the step_results list into one coherent reply that reflects what actually happened — no more, no less. The rules can be grouped into two themes:
  1. "Show the content" — early rules. If an agent returned a list of news items, the reply must include those items, not "Here are the news items:" followed by silence.
  2. "Don't lie" — the entire TRUTHFULNESS block. Three years of LLM development have shown that synthesizers will happily fabricate a "sent to Telegram" confirmation when no Telegram step ran. The rules explicitly forbid it and tell the model what to say instead.
The "Downloaded: line" rule exists because the synthesizer used to summarise file paths into descriptive prose ("I downloaded the image") — losing the actual path. Now it must copy the line verbatim, so the user (and any viga_agent step that comes next) can read the path.
The interim flag is set when one or more steps got offloaded to the background (e.g. a 3D render that takes minutes). The interim mode tells the synthesizer to acknowledge the background work explicitly so the user isn't left wondering.
