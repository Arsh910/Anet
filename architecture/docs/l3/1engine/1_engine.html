File 01 · AdaptOrch/ + engine_base.py

The AdaptOrch engine — the conductor.

The engine is AdaptOrchEngine(BaseEngine), split across anet/core/AdaptOrch/coordinator.py (the pipeline) and anet/core/engine_base.py (the shared infra). Every You: ... in your terminal hits engine.run_turn(thread_id, store, user_input) here and doesn't return until the synthesizer has streamed the answer back. It's a five-phase pipeline — decompose → DAG → route → execute → synthesize — one module per phase, plus a shared base. There is no longer a planner / executor / checker / synthesizer stage engine.

Where it sits

The engine is the top of the dependency tree — almost every other core module is something it calls. It uses the orchestrator (the agentic loop, one call per subtask), the orchestrator uses agent_runner, and so on.

Reading order recap: we explain the engine first because it's where execution starts. Each time it references another file (orchestrator, the memory backend, context_window, …) we'll cover that file in its own chapter — see the L3 index.

Why split into a package? Each phase is a distinct concern with a clean seam: the decomposer and synthesizer make LLM calls, the DAG and router are pure Python (no LLM, no I/O — easy to unit-test), and the executors are pure scheduling that call an injected run_subtask. coordinator.py is the thin file that threads them together; engine_base.py holds the state and helpers that any future engine would share. Plain Python throughout — no LangChain, no graph framework.
