What this tells you up front:
  • "Pure-Python replacement" — this file used to be a LangGraph StateGraph. It isn't anymore. There's no framework hiding control flow; everything you see runs in plain async/await.
  • "No LangGraph, no LangChain" — load-bearing in the README's "no framework lock-in" claim. Confirming it here in the file's own docstring matters: a future contributor who reaches for LangChain's RunnableSequence should pause.
  • "Messages are plain dicts" — the entire engine speaks one shape: {"role": "user"|"assistant", "content": str}. No Pydantic models, no custom classes. This is what makes the conversation cheap to serialise into conversations.db and easy to pass to any provider's SDK.

The third-party + intra-package imports lines 19–22

engine.py:19–22from openai import AsyncOpenAIfrom anet.core import orchestratorfrom anet.core.context import on_status as _status_var, on_token as _token_var, is_cancelled as _is_cancelled
Three imports, three distinct roles:
  • AsyncOpenAI — the only LLM SDK engine.py touches directly, used solely for the manager call (planner + synthesizer). Worker-agent calls go through orchestrator → agent_runner, which handles Anthropic and Vertex as well. The manager is always OpenAI-compatible (OpenRouter / Google / OpenAI / Vertex via OpenAI compat) — see _MANAGER_PROVIDERS in chapter 1b.
  • orchestrator — the agentic loop that runs one step. Engine.py never calls into a model directly for agent work; every step is shipped off to orchestrator.run(...). Covered fully in chapter 02.
  • on_status, on_token, is_cancelled — three ContextVar-style hooks defined in anet/core/context.py (chapter 16). They let the engine emit status updates ("manager: planning…") and token streams to whatever UI is listening, and check if the user pressed ESC, without the engine knowing anything about the UI. The underscore-prefix + as-rename makes it clear these are private-by-convention re-exports for use only inside this file.
The decoupling matters: these three context vars are the entire interface between engine and CLI. Swap the CLI for a web UI (the planned anet-server) and engine doesn't change a line — only the UI's on_status/on_token handlers do.

Two constants lines 24–25

engine.py:24–25_MAX_RETRIES    = 3_ASSISTANT_NAME = os.getenv("ASSISTANT_NAME", "Anet")
_MAX_RETRIES = 3 — the entire reason the engine can't run away. Every partial verdict from the checker bumps an attempts counter; when attempts >= 3 - 1 the step is forced to advance even if the verdict is still partial. We meet this constant again in chapter 1e's _check method, but it's declared here so the bound is visible at the top of the file. Changing it to 5 here would change the engine's patience globally — no other file needs editing.
_ASSISTANT_NAME — defaults to "Anet", but can be overridden with the ASSISTANT_NAME env var so a forked deployment can rebrand without touching code. It's injected into the planner and synthesizer prompts ("You are Anet, an AI assistant…").

Loading SOUL.md once lines 27–36

engine.py:27–36# ── Soul (loaded once at import, injected into manager prompts) ───────────────def _load_soul_once() -> str:    try:        from anet.core.config_loader import load_soul        return load_soul()    except Exception:        return ""_SOUL = _load_soul_once()
Why "loaded once": SOUL.md is the active pack's persona file — a few hundred bytes of free-form text that gets glued into the system prompt every turn. Reading the file from disk every _plan() call would be wasteful and would also make hot-reloading unpredictable. Reading it once at import time means:
  • It's loaded before run_turn ever runs (zero per-turn cost).
  • If SOUL.md doesn't exist (pack without one), load_soul() raises, we swallow the exception, and _SOUL = "" — every downstream check is just if _SOUL.
  • Switching packs via /changepack requires rebuilding the engine (which re-imports nothing — but reload of config is triggered). The CLI handles this; engine.py treats _SOUL as immutable for the lifetime of the engine instance.
The bare-except Exception looks lazy but is deliberate: this code runs at module import. If it raises, engine.py can't be imported, and the CLI dies before printing anything useful. A silent empty-string fallback is strictly better than a stack trace on startup when the pack hasn't been set up yet (e.g. on first run before ~/.anet/ is seeded).
Lazy import inside the function: from anet.core.config_loader import load_soul is inside _load_soul_once(), not at the top. This is the pattern we'll see again and again: top-of-file imports stay minimal (just stdlib + openai + orchestrator + the three context-vars), and anything that touches the workspace is imported lazily so circular imports and partially-initialised states never blow up engine import.
