the engine · section 1 of 3

BaseEngine.__init__ and the run_turn wiring.

The engine is two files. engine_base.py holds the shared state and infrastructure every engine inherits; coordinator.py holds AdaptOrchEngine.run_turn, which threads the five phases into one turn. This page reads the constructor, then walks the wiring that turns one user_input into a streamed reply.

The base-engine docstring engine_base.py:1–12

engine_base.py:1–12"""engine_base.py — shared engine infrastructure.Defines the BaseEngine class that every orchestration engine (AdaptOrchEnginetoday, future paper-based engines later) inherits from. Owns per-thread state(turn counters, rolling summaries), the rolling-summary maintenance, and thepersistence helper. Engine-specific pipelines (e.g. AdaptOrch's 5-phasedecompose → DAG → route → execute → synthesize) live in their respectivesubclasses; this file is engine-agnostic.Per-engine entry point: subclasses must implement `run_turn`."""
What this tells you up front:
  • "engine-agnostic" — the base owns nothing pipeline-specific. Per-thread state, the rolling-summary logic, and persistence live here; the actual orchestration (AdaptOrch's five phases) lives in the subclass. Swapping in a different orchestration strategy later means a new subclass, not a rewrite.
  • "subclasses must implement run_turn" — that's the one abstract method. BaseEngine.run_turn raises NotImplementedError; AdaptOrchEngine supplies the real one.
  • Messages are plain dicts — the whole engine speaks one shape: {"role": "user"|"assistant", "content": str}. No Pydantic models. That's what makes conversations cheap to serialise into conversations.db and easy to pass to any provider's SDK.

BaseEngine.__init__ — the state engine_base.py:86–99

engine_base.py:86–99def __init__(    self,    agents:        list[dict],    tools:         dict,    manager_tools: dict | None = None,) -> None:    self._agents       = agents    self._agent_map    = {a["name"]: a for a in agents}    self._tools        = tools    self._manager_tools = manager_tools or {}    # Incremental memory: turn counters per thread    self._turn_counts: dict[str, int] = {}    # Short-term memory: latest rolling summary per thread (for the agent path)    self._summaries: dict[str, str] = {}
What gets held, and why:
  • _agents — the list of agent configs (research / code / computer / checker, plus any pack-supplied agents). Each is a dict with a name, task_types, its model, and its toolbox.
  • _agent_map — the same agents keyed by name, built once. The coordinator's _resolve_agent uses it to turn a decomposer-assigned agent name into a live config in O(1).
  • _tools — the shared tool dict every subtask's agent draws from (each agent still only sees its own toolbox; this is the superset the orchestrator resolves against).
  • _manager_tools — tools the manager can call itself (rare; defaults to {}).
  • _turn_counts / _summaries — the only mutable per-thread state, both plain dicts keyed by thread_id. No database in the engine object itself: durable state lives in the store that's passed into run_turn.
No filesystem, no network in the constructor. Building an engine is cheap and side-effect-free — you can construct one before the workspace is fully set up. Everything that touches disk or a provider is lazy, inside the methods that need it.

The whole public API is one method. main.py calls result = await engine.run_turn(thread_id, store, user_input) and reads result.reply. Everything below is what AdaptOrchEngine does inside that call.

AdaptOrchEngine.run_turn — the outer guard coordinator.py:29–40

coordinator.py:29–40class AdaptOrchEngine(BaseEngine):    async def run_turn(self, thread_id, store, user_input) -> EngineResult:        try:            return await self._adaptorch_turn(thread_id, store, user_input)        except Exception as exc:            self._notify(f"manager: AdaptOrch error ({exc})")            reply = f"Sorry — something went wrong while handling that turn: {exc}"            try:                await self._persist(store, thread_id, user_input, reply, False)            except Exception:                pass            return EngineResult(reply=reply)
One try/except wraps the whole turn. A hard pipeline error — a cycle in the DAG, an unexpected exception inside a phase — surfaces as a clean error EngineResult rather than a crash. Note what changed from the old engine: there is no legacy fallback path any more. The turn either runs the five phases or reports a clean error; it is never silently re-routed to some other pipeline.

Before any phase runs, the engine builds the working context. _maintain_summary (on the base — chapter 1b) returns the rolling summary plus a keep_from cut point; render_for_prompt flattens "summary + recent verbatim turns" into a single string. exclude_last=True drops the current user_input because it's passed to the decomposer separately as the task — including it here would duplicate it.
This is what fixed AdaptOrch's amnesia. The working context threads prior-turn memory into both the decomposer (as memory_ctx) and every executor (as global_context), so a follow-up like "now do the same for the other repo" resolves correctly.

Phases 1–3 — decompose, DAG, route coordinator.py:60–74

coordinator.py:60–74# ── Phase 1 — Decomposition (+ trivial fast-path) ──────────────on_status("manager: decomposing task...")decomp = await decomposer.decompose(user_input, self._agents, memory_ctx=working_ctx)if decomp.trivial:    reply = decomp.reply or "Done."    await self._persist(store, thread_id, user_input, reply, False)    return EngineResult(reply=reply)# ── Phase 2 — DAG construction ─────────────────────────────────task_dag = decomposer.to_dag(decomp)   # raises CycleError on a cycle → caught above# ── Phase 3 — Topology routing ─────────────────────────────────decision = routermod.route(task_dag)on_status(f"manager: routed → {decision.topology.symbol} · {task_dag.summary()}")
Three phases, two of them pure.
  1. Decompose is the one LLM call so far — it returns a Decomposition that is either a trivial fast-path reply (greeting, one-liner) or a list of subtasks. A trivial decomposition short-circuits the whole pipeline: persist and return, no agents spun up.
  2. DAG (to_dag) is pure Python — it validates the subtasks and computes ω / δ / γ / layers. A cycle raises CycleError, caught by the outer guard.
  3. Route (router.route) is pure and O(1) — it reads the DAG's metrics and picks one of four topologies via Algorithm 1. The status line shows the chosen symbol (τ_P / τ_S / τ_H / τ_X) and the DAG shape.
The metrics and Algorithm 1's thresholds are covered in chapter 1b.

Phase 4 — execution, wired to the agent loop coordinator.py:76–97

coordinator.py:76–97# ── Phase 4 — Execution ────────────────────────────────────────async def run_subtask(st, prompt: str) -> StepResult:    agent = self._resolve_agent(st.agent)    try:        result = await orchestrator.run(agent, self._tools, prompt, [], on_status)        text = (result or {}).get("text", "")        return StepResult(id=st.id, agent=agent["name"], description=st.description, output=text)    except Exception as exc:        return StepResult(id=st.id, agent=agent["name"], description=st.description,                          output=f"[subtask failed: {exc}]", success=False, error=str(exc))ctx = ExecContext(run_subtask=run_subtask, global_context=working_ctx, on_status=on_status)async def reroute_fn(gamma_override: float):    d2 = routermod.route(task_dag, gamma_override=gamma_override)    on_status(f"manager: re-routing → {d2.topology.symbol} (γ'={gamma_override:.2f})")    return d2.topology, await execute(d2.topology, task_dag, ctx)results = await execute(decision.topology, task_dag, ctx)
This is the seam between the engine and the agent loop. The executors (chapter 1b) are pure scheduling — they know a topology and how to assemble each subtask's prompt, but they don't know about agents or tools. The engine injects run_subtask: it resolves the decomposer-assigned agent name to a live config and ships the assembled prompt to orchestrator.run — the agentic loop, one run per subtask (chapter 02). The result is wrapped in a StepResult; a raised exception becomes a success=False result rather than aborting the whole turn.
reroute_fn is handed to synthesis, not called here. It re-routes with an inflated coupling estimate and re-executes — but only Phase 5's resolve operator ever invokes it, and only on a genuine contradiction. See below.

Phase 5 — synthesis, then log + persist coordinator.py:99–129

coordinator.py:99–129 · excerpt# ── Phase 5 — Adaptive synthesis ───────────────────────────────on_status("manager: synthesizing...")synth = await synthesizer.synthesize(    results, topology=decision.topology, task=user_input,    dag=task_dag, request_class=decomp.request_class, reroute_fn=reroute_fn,)reply = synth.output or "I worked through the task but produced no final answer."# Log the routing decision + outcome (the learned-routing dataset).usage = tokens.current()routermod.log_route(decision, request_class=decomp.request_class, outcome={    "method": synth.method, "consistency": round(synth.consistency, 3),    "iterations": synth.iterations, "rerouted": synth.rerouted, ...})await self._persist(store, thread_id, user_input, reply, False)step_results = [{"agent": r.agent, "status": "ok" if r.success else "failed",                "result": r.output} for r in results]return EngineResult(reply=reply, step_results=step_results)
Synthesis is Algorithm 2 — a dispatcher over operators, not a gate. synthesize picks the operator that fits the topology + request class (compose / aggregate / vote / rank / resolve), streams the combined answer to your terminal, and returns a SynthesisResult. The reroute_fn from Phase 4 is passed in but is used only by resolve, on a real contradiction. Consistency score is just a signal inside vote/resolve — it is no longer a global merge/arbiter gate. The operator prompts are in chapter 1c.
Every decision is logged. router.log_route appends the DAG features, chosen topology, and the synthesis outcome (operator, consistency, iterations, tokens) to routing_log.jsonl — the training signal for future learned routing. It's collected from the first decision, not bolted on later.

_resolve_agent — the one coordinator helper coordinator.py:133–140

coordinator.py:133–140def _resolve_agent(self, name: str) -> dict:    if name and name in self._agent_map:        return self._agent_map[name]    if "code_agent" in self._agent_map:        return self._agent_map["code_agent"]    return self._agents[0]
A bad or empty agent assignment never crashes. The decomposer assigns an agent name per subtask; if it names one that doesn't exist (or leaves it blank), this falls back to code_agent, then to the first agent. The _agent_map built in the constructor makes the lookup O(1).
